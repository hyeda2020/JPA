# 2. 엔티티 매핑
- 객체와 테이블 매핑
  - `@Entity` : 해당 어노테이션이 붙은 클래스는 JPA가 관리하는 엔티티
    - 단, 해당 클래스는 기본 생성자 필수
    - final 클래스 및 enum, interface, inner 클래스는 엔티티로 사용 불가
    - DB에 저장할 필드에는 final 사용 불가
  - `@Table` : 엔티티와 매핑할 테이블 지정
    - 속성

- 데이터베이스 스키마 자동 생성
  - JPA는 애플리케이션 로딩 시점에 DDL을 자동 생성하여 실행(여러 DB 언어 지원)
  - 단, 이런 DDL 자동 생성 기능은 개발 장비에서만 사용해야 함!
  - 제약 조건
  ```java
  // DDL 실행 시 해당 필드의 유니크 제약조건 및 사이즈 설정
  @Column(unique = true, length = 10)
  private String name;  
  ```

- 필드와 컬럼 매핑
  - `@Column` : 컬럼 매핑
    - name : 필드와 매핑할 테이블의 컬럼 이름
    - insertable, updatable : 등록, 변경 가능 여부(기본값 : TRUE)
    - nullable(DDL) : null 값의 허용 여부를 설정  
      (false로 설정하면 DDL 생성 시에 not null 제약조건이 붙음)
    - unique(DDL) : @Table의 `uniqueConstraints`와 같지만 한 컬럼에 간단히 유니크 제약조건을 걸 때 사용
    - columnDefinition(DDL) : 데이터베이스 컬럼 정보를 직접 제공  
      ex) varchar(100) default ‘EMPTY'
    - length(DDL) : 문자 길이 제약조건, String 타입에만 사용(기본값 : 255)
    - precision, scale(DDL) : BigDecimal 타입에서 사용(기본값 : precision=19, scale=2)
  - `@Enumerated` : 자바 enum 타입을 매핑(기본값 : EnumType.ORDINAL)
    - `EnumType.ORDINAL`: enum 순서를 데이터베이스에 저장  
      **=> 단, 주의할 점은 ORDINAL의 경우 DB에 저장되는 순서 값과 Enum 정의 순서 값이 달라질 수 있으므로 실무에서는 절대 사용 금지!**
    - `EnumType.STRING`: enum 이름을 데이터베이스에 저장
  - `@Temporal` : 날짜 타입(java.util.Date, java.util.Calendar)을 매핑할 때 사용
    - `TemporalType.DATE`: 날짜, 데이터베이스 date 타입과 매핑  
      ex) 2013–10–11
    - `TemporalType.TIME`: 시간, 데이터베이스 time 타입과 매핑  
      ex) 11:11:11
    - `TemporalType.TIMESTAMP`: 날짜와 시간, 데이터베이스 timestamp 타입과 매핑  
      ex) 2013–10–11 11:11:11
  - `@Lob` : 데이터베이스 BLOB, CLOB 타입과 매핑
  - `@Transient` : 해당 필드는 데이터베이스에 매핑하지 않음
  
  
- 기본 키 매핑
  - 직접 할당 : `@Id` 만 사용
  - 자동 생성 : `@GeneratedValue`
    - `IDENTITY` : 기본 키 생성을 데이터베이스에 위임
      1) JPA는 보통 트랜잭션 커밋 시점에 INSERT SQL 실행
      2) `AUTO_ INCREMENT`는 데이터베이스에 INSERT SQL을 실행한 이후에 ID 값을 알 수 있음
      3) 하지만 엔티티를 영속성 컨텍스트에 저장하기 위해선 반드시 PK가 있어야 함.
      4) 때문에 `IDENTITY` 전략은 `em.persist()` 시점에 즉시 INSERT SQL 실행하고 DB에서 식별자를 조회하여 엔티티에 부여
      ```
      @Entity
      public class Member {
          @Id
          @GeneratedValue(strategy = GenerationType.IDENTITY)
          private Long id;
          ...
      }
      ```
    - `SEQUENCE` : 데이터베이스 시퀀스 오브젝트 사용
      -> `@SequenceGenerator` 필요
      1) `allocationSize`는 50~100 으로 최적화 세팅하는 것이 권장
      2) 만약, 1로 세팅하면 값이 증가할 때마다 DB호출을 해야되지만, 
      3) 50으로 세팅하면 DB를 한 번 호출하여 시퀀스 값을 51로 세팅해 놓고
      4) WAS 내 메모리에서 1~50까지 사용한 후 그 다음번에 DB를 호출해도 되므로 효율적(DB서버 호출 최소화)
      ```
      @Entity
      @SequenceGenerator(
          name = “MEMBER_SEQ_GENERATOR", // 식별자 생성기 이름
          sequenceName = “MEMBER_SEQ",   // 매핑할 데이터베이스 시퀀스 이름
          initialValue = 1,              // 시퀀스 DDL을 생성할 때 처음 시작하는 수를 지정
          allocationSize = 50)           // 시퀀스 한 번 호출에 증가하는 수(DB 시퀀스 값이 하나씩 증가하도록 설정되어 있으면 반드시 1로 세팅)
      public class Member {
          @Id
          @GeneratedValue(strategy = GenerationType.SEQUENCE,
                          generator = "MEMBER_SEQ_GENERATOR")
          private Long id;
          ...
      }
      ```
    - `TABLE` : 키 생성용 테이블을 활용하는 것으로 모든 DB에서 사용 가능하나, 성능 하락
      -> `@TableGenerator` 필요
      ```
      @Entity
      @TableGenerator(
          name = "MEMBER_SEQ_GENERATOR", // 식별자 생성기 이름
          table = "MY_SEQUENCES",        // 키 생성 테이블명
          pkColumnValue = “MEMBER_SEQ",  // 시퀀스 컬럼명
          allocationSize = 1)            // 시퀀스 한 번 호출에 증가하는 수
      public class Member {
          @Id
          @GeneratedValue(strategy = GenerationType.TABLE,
                          generator = "MEMBER_SEQ_GENERATOR")
          private Long id;
          ...
      }
      ```
    - `AUTO` : DB 방언에 따라 자동 지정(기본값)
  - 권장하는 식별자 전략
    - 기본 키 제약 조건: NOT NULL, UNIQUE, 변동성 X
    - 비즈니스 로직 요소(전화번호, 주민등록번호 등)은 식별자로 사용하지 않기
    - 임의 생성된 값 등 비즈니스와 관련 없는 대체키 활용
    - 권장 : Long형 + 대체키 + 키 생성전략 사용
